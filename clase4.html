git en vscode


usando git en vscode

comandos en vscode desde la terminal

  • git revert
    • si queremos revertir algun commit en la linea cronologica primero nos fijamos en el codigo del commit, lo copiamos en git revert y listo, me devuelvo hasta ese commit. esto crea un nuevo commit. depende del proyecto en el que estemos o la empresa, vamos a usar este o no. al usar esto deja el historial 'sucio', pero de cierta manera es bueno porque permite ver el historial de los cambios que hicimos en cambio al usar reset deja un historial mas limpio porque permite borrar commits.
  • git reset
    • elimina los commit de la linea del historial, siempre y cuando sean locales. existen tres tipos de reset. git reset 'tipo de reset'
      1. --soft conserva los cambios y los manda al stage area.
      2. --mixed conserva los cambios, y te los deja en el working directory y no los manda al stage area.
      3. --hard si vuelvo a un commit antiguo me lleva alli pero los commits que hice desde donde estaba hasta donde fui pierdo los cambios.

    para hacer un push tengo que estar parado en el ultimo commit del proyecto en el cliente de git... es para asegurarse de que todos los colaboradores estamos actualizados con los cambios...


  • git blame
    • es para ver los cambios que hizo alguien en el proyecto, asi se puede ver quien se mando una macana. git blame codigo del commit. git blame nombre del commit. te tira el codigo, el usuario de github la fecha y hora en que se hizo.
  • git blame -w
    • muestra los archivos que no se commitearon y que estan los espacios en blanco.
  • git grep
    • es para hacer una busqueda, busca una palabra en donde o especifique. si no le pongo una palabra clave entonces busca... es como el ctrl+F en vs code. es util para buscar algo en un repositorio.
    • git grep -n 'palabraclave'
    • git grep -c 'palabraclave'
    • git grep

flujo de trabajo


cualquier empresa tiene su flujo de trabajo. los siguientes son unos ejemplos

  • long-running branch
    • consta de 3 ramas main, testing y la del desarrollador. es un flujo clasico y simple, pero no es escalable.
  • topic branch
    • son ramas cortita, para un tema en particular por mas peque;o que sea, o hasta para las ideas para solucionar cualquier problema. y al final se mergea al main y luego se borra. y al final la rama que se escoja se mergea.
  • admin de integracion(similar al de meli)
    • un repo que lo tiene un admin y el equipo hace fork del repo para trabajar localmente, luego se hace un pull request y el admin es el que aprueba o rechaza el PR para mergear...
  • dictador y tenientes
    • estructura de trabajo jerargica que se usa en un equipo de mas de mil personas...
  • gitflow(se usa en meli, y el mas usado)
    • se crean dos ramas, la main y la develop(o cualquier nombre que se crea), y todos los cambios que se hacen se hacen en la rama develop... se trabaja con release... cambios que se hacen cada cierto tiempo... la master se usa solo para hacer los releases... y los features que se hagan en su respectivo sprint se mergean en la rama develop... y antes de pasar a produccion se hace una rama de mantenimiento en donde se hace el testing ... el hotfix es una rama de emergencia para hacer un cambio de emergencia...
